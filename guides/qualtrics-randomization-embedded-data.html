1. What this guide is about

This guide explains how to randomize experimental conditions in Qualtrics when using SMSLab stimuli.

The focus is on a practical workflow:

  1. Create one stimulus version per condition.
  2. Create one Qualtrics block per condition.
  3. Add the corresponding HTML stimulus inside each block.
  4. Add Embedded Data fields to record the assigned condition.
  5. Use the Qualtrics Randomizer to assign participants to one condition.
  6. Add the interaction Embedded Data fields generated by SMSLab.
  7. Test that both condition assignment and interaction tracking are correctly saved.

The main idea is simple: Qualtrics should know both what condition the participant saw and what the participant did inside the stimulus.

2. Randomization vs. tracking

When using SMSLab stimuli in Qualtrics, you usually deal with two different types of information.

The first is condition information. This tells you which experimental condition the participant was assigned to.

Example:

condition = AI_Disclosure
disclosure = AI
source = Brand

The second is interaction information. This tells you what the participant did inside the stimulus.

Example:

__js_igfeed_post1_like = 1
__js_igfeed_post1_comment = 0
__js_igfeed_post1_save = 1

These two types of information should be stored separately.

Condition Embedded Data is created by you in Qualtrics.

Interaction Embedded Data is generated by SMSLab and should be declared in Qualtrics Survey Flow.

Qualtrics Survey Flow allows you to add elements such as Embedded Data, Randomizers, Branches, and Groups; the Randomizer can randomly present blocks, Embedded Data, and other Survey Flow elements to respondents. (Qualtrics)

3. The basic structure

A clean SMSLab experiment in Qualtrics usually has this structure:

Survey Flow

Embedded Data: interaction fields

  __js_igfeed_post1_like
  __js_igfeed_post1_comment
  __js_igfeed_post1_save
  ...

Randomizer: randomly present 1 of the following elements

  Group: Condition 1
    Embedded Data: condition = condition_1
    Block: Stimulus Condition 1
  Group: Condition 2
    Embedded Data: condition = condition_2
    Block: Stimulus Condition 2
  • Outcome measures
  • Manipulation checks
  • Demographics
  • End of survey

This structure does three things:

  1. It prepares the interaction fields before the participant sees the stimulus.
  2. It randomly assigns the participant to one condition.
  3. It saves the assigned condition in the dataset.

The Group element is useful because it keeps together the condition Embedded Data and the block containing the stimulus. Qualtrics supports Groups in Survey Flow to group different Survey Flow elements together. (Qualtrics)

4. Why each condition should usually be a group

For each experimental condition, you should create a group that contains:

  1. The Embedded Data values identifying that condition.
  2. The block containing the corresponding SMSLab stimulus.
This is important because the condition variable must be set before or together with the stimulus block assigned to the participant.

For example:

Group: Condition A

  Embedded Data:
    condition = A
  Block:
    Instagram feed stimulus - Condition A

For a simple two-condition experiment:

Group: Control

  Embedded Data:
    condition = control
  Block:
    Control feed

Group: Treatment

  Embedded Data:
    condition = treatment
  Block:
    Treatment feed

Then both groups go inside the Randomizer.

5. Example 1: simple two-condition experiment

Imagine a study testing whether an AI disclosure affects consumers’ perception of a social media post.

You have two conditions:

  • Condition 1: No AI disclosure
  • Condition 2: AI disclosure

You should create two feed versions in SMSLab:

  • Feed 1: No AI disclosure
  • Feed 2: AI disclosure

Then, in Qualtrics, create two blocks:

  • Block: Feed - No AI disclosure
  • Block: Feed - AI disclosure

Inside each block, add a Descriptive Text question and paste the corresponding generated HTML stimulus.

Your Survey Flow should look like this:

Embedded Data: SMSLab interaction fields

  __js_igfeed_post1_like
  __js_igfeed_post1_comment
  __js_igfeed_post1_save
  __js_igfeed_post1_ai_info_opened
  ...

Randomizer: randomly present 1 of the following elements

  Group: No AI disclosure
    Embedded Data:
      condition = no_ai_disclosure
      ai_disclosure = no
    Block:
      Feed - No AI disclosure
  Group: AI disclosure
    Embedded Data:
      condition = ai_disclosure
      ai_disclosure = yes
    Block:
      Feed - AI disclosure
Block:
  Manipulation checks
Block:
  Dependent variables

In the final dataset, a participant assigned to the AI disclosure condition may have:

condition = ai_disclosure
ai_disclosure = yes
__js_igfeed_post1_like = 1
__js_igfeed_post1_comment = 0
__js_igfeed_post1_ai_info_opened = 1

This tells you that the participant saw the AI disclosure condition and opened the AI Information label.

6. Example 2: 2 × 2 factorial design

Now imagine a 2 × 2 experiment.

The factors are:

  • Disclosure: AI disclosure vs No AI disclosure
  • Engagement: High engagement vs Low engagement

This creates four conditions:

  • Condition 1: AI disclosure + High engagement
  • Condition 2: AI disclosure + Low engagement
  • Condition 3: No AI disclosure + High engagement
  • Condition 4: No AI disclosure + Low engagement

You should create four stimulus versions in SMSLab:

  • Feed 1: AI disclosure + High engagement
  • Feed 2: AI disclosure + Low engagement
  • Feed 3: No AI disclosure + High engagement
  • Feed 4: No AI disclosure + Low engagement

Then create four Qualtrics blocks:

  • Block: AI disclosure - High engagement
  • Block: AI disclosure - Low engagement
  • Block: No AI disclosure - High engagement
  • Block: No AI disclosure - Low engagement

Each block contains the corresponding HTML stimulus.

In Survey Flow, use one Randomizer that presents one of four groups:

Embedded Data: SMSLab interaction fields

  __js_igfeed_post1_like
  __js_igfeed_post1_comment
  __js_igfeed_post1_save
  __js_igfeed_post1_carousel_second_seen
  ...

Randomizer: randomly present 1 of the following elements

  Group: AI disclosure + High engagement
    Embedded Data:
      condition = AI_High
      disclosure = AI
      engagement = High
    Block:
      AI disclosure - High engagement
  Group: AI disclosure + Low engagement
    Embedded Data:
      condition = AI_Low
      disclosure = AI
      engagement = Low
    Block:
      AI disclosure - Low engagement
  Group: No AI disclosure + High engagement
    Embedded Data:
      condition = NoAI_High
      disclosure = NoAI
      engagement = High
    Block:
      No AI disclosure - High engagement
  Group: No AI disclosure + Low engagement
    Embedded Data:
      condition = NoAI_Low
      disclosure = NoAI
      engagement = Low
    Block:
      No AI disclosure - Low engagement

This creates a clean dataset because each participant has:

condition
disclosure
engagement
  • interaction variables
  • dependent variables

For example:

condition = AI_High
disclosure = AI
engagement = High
__js_igfeed_post1_like = 1
__js_igfeed_post1_save = 0

This is better than storing only a single condition variable, because separate factor variables make analysis easier.

7. Example 3: 2 × 3 design

Suppose you have a 2 × 3 design.

The factors are:

  • Disclosure: AI vs Human
  • Message tone: Emotional vs Functional vs Neutral

This creates six conditions:

  • AI + Emotional
  • AI + Functional
  • AI + Neutral
  • Human + Emotional
  • Human + Functional
  • Human + Neutral

In this case, create six SMSLab stimuli and six Qualtrics blocks.

Your condition Embedded Data should include:

condition
disclosure
tone

For example:

Group: AI + Emotional

  Embedded Data:
    condition = AI_Emotional
    disclosure = AI
    tone = Emotional
  Block:
    Feed - AI Emotional

Another group:

Group: Human + Functional

  Embedded Data:
    condition = Human_Functional
    disclosure = Human
    tone = Functional
  Block:
    Feed - Human Functional

This makes the final dataset easier to use in ANOVA, regression, PROCESS, SEM, or other analyses.

8. Why factor-level Embedded Data are useful

For factorial designs, do not rely only on a single condition field.

You can save:

condition = AI_High

but it is also useful to save:

disclosure = AI
engagement = High

This helps during analysis because you can directly test main effects and interactions.

For a 2 × 2 design, recommended condition fields are:

condition
  • factor1
  • factor2

Example:

condition = AI_High
disclosure = AI
engagement = High

For a 2 × 2 × 2 design, use:

condition
  • factor1
  • factor2
  • factor3

Example:

condition = AI_High_Influencer
disclosure = AI
engagement = High
source = Influencer

This reduces the risk of confusion during data cleaning.

9. Where to place interaction Embedded Data fields

The interaction fields generated by SMSLab should usually be placed before the Randomizer.

Example:

Embedded Data: SMSLab interaction fields

  __js_igfeed_post1_like
  __js_igfeed_post1_comment
  __js_igfeed_post1_save
Randomizer
  Group: Condition 1
  Group: Condition 2

The fields should be declared before the stimulus appears, so the JavaScript inside the stimulus can write values to them.

Leave the values blank.

Do not manually set these fields to 0 in Survey Flow. The generated stimulus code initializes and updates them during the survey.

Qualtrics documentation explains that adding Embedded Data in Survey Flow allows information to be saved in survey results; therefore, the interaction fields need to exist in Survey Flow if you want them to appear in the dataset. (Qualtrics)

10. Where to place condition Embedded Data fields

Condition Embedded Data should be placed inside each condition group, before the block containing the stimulus.

Example:

Group: AI disclosure

  Embedded Data:
    condition = AI
    disclosure = yes
  Block:
    AI disclosure feed

This ensures that when a participant is assigned to that group, Qualtrics saves the correct condition values.

Do not place all condition values at the top of the Survey Flow. If you set:
condition = AI
condition = NoAI

globally, Qualtrics will not know which one should apply to which participant. The condition values must be assigned inside the randomized group.

This structure is simple, transparent, and easy to analyze.

12. Step-by-step: creating the structure in Qualtrics

Step 1: create the stimulus blocks

In Qualtrics, create one block for each condition.

Example:

  • Stimulus - Condition A
  • Stimulus - Condition B
  • Stimulus - Condition C
  • Stimulus - Condition D

Inside each block:

  1. Add a Descriptive Text question.
  2. Open HTML/source view.
  3. Paste the HTML generated by SMSLab for that condition.
  4. Save the question.

Step 2: open Survey Flow

Go to:

Survey Flow

Survey Flow is where Qualtrics lets you organize blocks and add logic such as Embedded Data, Randomizers, Branches, and Groups. (Qualtrics)

Step 3: add interaction Embedded Data

At the beginning of Survey Flow, add an Embedded Data element.

Insert all interaction fields shown by SMSLab.

Example:

__js_igfeed_post1_like
__js_igfeed_post1_comment
__js_igfeed_post1_comment_submitted
__js_igfeed_post1_comment_text
__js_igfeed_post1_save

Leave all values blank.

Step 4: add a Randomizer

Below the interaction Embedded Data element, add a Randomizer.

Set it to:

Randomly present 1 of the following elements

Qualtrics’ Randomizer can randomly present a selected number of nested elements and includes an option to evenly present elements across respondents. (Qualtrics)

Step 5: create one group per condition

Inside the Randomizer, create one Group for each condition.

Each group should contain:

  1. Embedded Data identifying the condition.
  2. The stimulus block for that condition.

Example:

Group: Condition A

  Embedded Data:
    condition = A
  Block:
    Stimulus - Condition A

Step 6: add the rest of the survey after the Randomizer

After the Randomizer, add the rest of the questionnaire:

  • Manipulation checks
  • Mediators
  • Dependent variables
  • Controls
  • Demographics
  • Attention checks
  • Debriefing

Usually, these blocks should be outside the Randomizer so that all participants answer the same post-stimulus questions.

13. One-factor design example

Suppose your study manipulates message framing:

Emotional framing vs Functional framing

You need two conditions.

Survey Flow:

Embedded Data: interaction fields

  __js_igfeed_post1_like
  __js_igfeed_post1_save
  __js_igfeed_post1_comment

Randomizer: present 1 of 2

  Group: Emotional framing
    Embedded Data:
      condition = emotional
      framing = emotional
    Block:
      Stimulus - Emotional framing
  Group: Functional framing
    Embedded Data:
      condition = functional
      framing = functional
    Block:
      Stimulus - Functional framing
Block:
  Outcome measures

Dataset example:

condition = emotional
framing = emotional
__js_igfeed_post1_like = 1
brand_attitude = 5.8
purchase_intention = 4.9

14. Two-factor design example

Suppose your study manipulates:

  • Source: Brand vs Influencer
  • Disclosure: Present vs Absent

This is a 2 × 2 design.

Create four condition groups:

  • Brand + Disclosure
  • Brand + No Disclosure
  • Influencer + Disclosure
  • Influencer + No Disclosure

Survey Flow:

Embedded Data: interaction fields

  __js_igfeed_post1_like
  __js_igfeed_post1_comment
  __js_igfeed_post1_save

Randomizer: present 1 of 4

  Group: Brand + Disclosure
    Embedded Data:
      condition = brand_disclosure
      source = brand
      disclosure = present
    Block:
      Stimulus - Brand Disclosure
  Group: Brand + No Disclosure
    Embedded Data:
      condition = brand_no_disclosure
      source = brand
      disclosure = absent
    Block:
      Stimulus - Brand No Disclosure
  Group: Influencer + Disclosure
    Embedded Data:
      condition = influencer_disclosure
      source = influencer
      disclosure = present
    Block:
      Stimulus - Influencer Disclosure
  Group: Influencer + No Disclosure
    Embedded Data:
      condition = influencer_no_disclosure
      source = influencer
      disclosure = absent
    Block:
      Stimulus - Influencer No Disclosure

Dataset example:

condition = influencer_disclosure
source = influencer
disclosure = present
__js_igfeed_post1_like = 0
__js_igfeed_post1_save = 1

This makes it easy to test:

  • main effect of source
  • main effect of disclosure
  • source × disclosure interaction

15. Three-factor design example

Suppose your study manipulates:

  • Source: Brand vs Influencer
  • Disclosure: Present vs Absent
  • Engagement: High vs Low

This is a 2 × 2 × 2 design, meaning eight conditions.

You should create eight groups inside the Randomizer.

Each group should include:

condition
source
disclosure
engagement

Example group:

Group: Brand + Disclosure + High engagement

  Embedded Data:
    condition = brand_disclosure_high
    source = brand
    disclosure = present
    engagement = high
  Block:
    Stimulus - Brand Disclosure High

Another group:

Group: Influencer + No disclosure + Low engagement

  Embedded Data:
    condition = influencer_no_disclosure_low
    source = influencer
    disclosure = absent
    engagement = low
  Block:
    Stimulus - Influencer No Disclosure Low

In the dataset, each participant will have clear factor-level variables:

condition = influencer_no_disclosure_low
source = influencer
disclosure = absent
engagement = low

This is much easier to analyze than trying to recode everything from a single condition name.

16. Should the Embedded Data prefix change across conditions?

Usually, no.

If each participant sees only one feed, keep the same SMSLab prefix across conditions.

Example:

igfeed

This produces the same interaction variable names for all participants:

__js_igfeed_post1_like
__js_igfeed_post1_save
__js_igfeed_post1_comment

Then use separate condition variables to know which feed the participant saw:

condition
source
disclosure
engagement

This is usually cleaner because all participants share the same interaction columns.

Change the prefix only if the same participant sees more than one feed and you need separate interaction variables for each feed.

Example:

feed1
feed2

This would produce:

__js_feed1_post1_like
__js_feed2_post1_like

17. Between-subjects vs. within-subjects designs

Between-subjects design

In a between-subjects design, each participant sees only one condition.

Recommended structure:

Randomizer: present 1 condition group

Use the same interaction prefix across conditions.

Save the assigned condition with Embedded Data inside each condition group.

Within-subjects design

In a within-subjects design, each participant sees multiple conditions or stimuli.

In this case, you may need:

  • different prefixes for different feeds
  • randomized order of blocks
  • order variables
  • counterbalancing

For example:

  • feed1_post1_like
  • feed2_post1_like

You may also save order variables:

first_feed = emotional
second_feed = functional

Within-subjects designs require more careful planning because interaction fields must distinguish which stimulus each behavior belongs to.

18. Randomizing order vs. randomizing assignment

There are two different randomization goals.

Randomizing assignment

This means assigning each participant to one condition.

Example:

Participant sees only one of four conditions.

Use:

Randomizer: present 1 of 4 groups

Randomizing order

This means every participant sees multiple blocks, but the order changes.

Example:

Every participant sees Feed A and Feed B, but the order is randomized.

Use:

Randomizer: present all elements in random order

or use multiple Randomizers/Groups depending on the design.

Do not confuse these two procedures. For most between-subjects experiments, you want random assignment, not random order.

19. Interaction fields for conditions with different numbers of posts

Ideally, all conditions should have the same number of posts and the same target post position.

However, if conditions have different numbers of posts, the interaction fields may differ.

Example:

  • Condition A has 3 posts
  • Condition B has 5 posts

Then Condition B may produce fields such as:

__js_igfeed_post5_like

while Condition A will not.

This is not necessarily wrong, but it makes the dataset harder to analyze.

Recommended approach:

Keep the same number of posts across conditions.

Keep the target post in the same position.

Keep the same interaction field structure across conditions.

20. Naming conventions

Use short, clean, consistent names.

Good condition names:

  • AI_High
  • AI_Low
  • NoAI_High
  • NoAI_Low

Good factor names:

ai_disclosure
engagement
source
tone
framing

Avoid names with spaces or unusual characters.

Instead of:

AI disclosure + high engagement

use:

AI_HighEngagement

or:

AI_High

Clean naming reduces mistakes during analysis.

21. Testing the randomization

Before launching, test your randomization.

Recommended procedure:
  1. Preview the survey several times.
  2. Check that different conditions appear across previews.
  3. Complete at least one test response per condition.
  4. Export or inspect the response data.
  5. Confirm that the condition Embedded Data matches the stimulus shown.
  6. Confirm that interaction fields are saved.

For example, if a participant sees the “Brand + Disclosure” stimulus, the dataset should show:

condition = brand_disclosure
source = brand
disclosure = present

If this does not match, the Embedded Data is probably placed in the wrong group.

22. Common mistakes

Mistake 1: putting all condition values at the top of Survey Flow

Wrong:

Embedded Data:
  condition = A
  condition = B
  condition = C

This does not assign participants correctly.

Correct:

Randomizer
  Group A
    Embedded Data: condition = A
    Block: Stimulus A
  Group B
    Embedded Data: condition = B
    Block: Stimulus B

Mistake 2: not grouping condition data with the stimulus block

Wrong:

Randomizer
  Embedded Data: condition = A
  Embedded Data: condition = B
  Block: Stimulus A
  Block: Stimulus B

This can randomize condition labels and stimulus blocks separately, creating mismatches.

Correct:

Randomizer
  Group A
    Embedded Data: condition = A
    Block: Stimulus A
  Group B
    Embedded Data: condition = B
    Block: Stimulus B

Mistake 3: using different prefixes for no reason

This creates unnecessary complexity.

If each participant sees one feed, keep:

igfeed

for all conditions.

Mistake 4: forgetting factor-level variables

For factorial designs, do not save only:

condition = A

Also save the factors:

source = brand
disclosure = present

This makes analysis easier.

Mistake 5: declaring interaction fields after the stimulus

Interaction Embedded Data fields should be declared before the stimulus block.

If they are placed after the stimulus, values may not be saved correctly.

Here is the complete workflow.

  • Step 1: define the factors

  present
  absent

brand_present
brand_absent
influencer_present
influencer_absent
  • Step 3: create four Qualtrics blocks
  • Stimulus - Brand Present
  • Stimulus - Brand Absent
  • Stimulus - Influencer Present
  • Stimulus - Influencer Absent
  • Step 4: paste the corresponding HTML into each block

Use one Descriptive Text question per stimulus block.

Step 5: add interaction Embedded Data before the Randomizer

__js_igfeed_post1_like
__js_igfeed_post1_comment
__js_igfeed_post1_save

...

Step 6: create a Randomizer

Set it to present one group.

  • Step 7: create one group per condition
  • Group: Brand Present
  Embedded Data:
    condition = brand_present

    Stimulus - Brand Present

24. Suggested reporting in a methods section

You can describe the randomization procedure like this:

Participants were randomly assigned to one of four experimental conditions using the Qualtrics Survey Flow Randomizer. Each condition was implemented as a separate Survey Flow group containing condition-specific Embedded Data fields and the corresponding stimulus block. The condition Embedded Data recorded the assigned levels of the manipulated factors. Interaction data generated by the HTML stimulus were saved separately through Qualtrics Embedded Data fields.

For a 2 × 2 design:

The study used a 2 × 2 between-subjects design. The manipulated factors were source type and disclosure. In Qualtrics, each of the four experimental conditions was implemented as a separate randomized group. Each group contained Embedded Data fields identifying the assigned condition, source type, and disclosure level, followed by the corresponding SMSLab-generated HTML feed stimulus.

25. Final checklist

Before launching, check that:

Each condition has its own stimulus block.

Each condition has its own group in Survey Flow.

Each group contains condition Embedded Data.

Each group contains the correct stimulus block.

The Randomizer presents only one condition group.

Interaction Embedded Data fields are declared before the Randomizer.

The same SMSLab prefix is used across conditions when each participant sees only one feed.

Factor-level variables are saved for factorial designs.

Post-stimulus measures are outside the Randomizer.

The full survey has been tested.

The dataset correctly records both condition assignment and interaction variables.

A good Qualtrics setup should make it clear what the participant saw and what the participant did. This is the key principle behind combining SMSLab stimuli, condition Embedded Data, and interaction tracking.
